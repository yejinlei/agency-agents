## 1. 战略基础

### 1.1 NEXUS 解决什么

单个代理很强大。但如果没有协调，它们会产生：
- 冲突的架构决策
- 跨部门的重复努力
- 交接边界的质量缺口
- 无共享上下文或制度记忆

**NEXUS 通过定义以下内容消除这些失败模式：**
- **谁**在每个阶段激活
- **什么**他们产出以及为谁
- **何时**他们交接以及交给谁
- **如何**在推进前验证质量
- **为什么**每个代理存在于管线中（无乘客）

### 1.2 核心原则

| 原则 | 描述 |
|-----------|-------------|
| **管线完整性** | 任何阶段在未通过质量关卡前不会推进 |
| **并行执行** | 多个代理同时工作，通过标准化交接协议协调 |
| **质量关卡** | 每个阶段有可验证的通过/失败标准 |
| **证据驱动** | 所有决策必须有证据支持，非直觉 |
| **最小认知负荷** | 代理接收明确、精简的上下文，非整个代码库 |

### 1.3 成功定义

NEXUS 成功当：
- 所有代理在定义的时间线内完成其产出
- 质量关卡在推进前捕获所有缺陷
- 交接无上下文丢失
- 最终产品满足或超过规格要求
- 代理网络可扩展到新项目而无重新设计

## 2. NEXUS 运营模式

### 2.1 三种激活模式

| 模式 | 代理数 | 时间线 | 适用场景 |
|------|--------|--------|----------|
| **NEXUS-Full** | 全部 | 12-24 周 | 完整产品，从零构建 |
| **NEXUS-Sprint** | 15-25 | 2-6 周 | 功能开发，MVP |
| **NEXUS-Micro** | 5-10 | 1-5 天 | 特定任务，修复，审计 |

### 2.2 代理激活顺序

**全模式激活序列：**

```
第 0 阶段：情报与发现
  → 产品负责人（规格细化）
  → 市场研究专家（竞争分析）

第 1 阶段：战略与架构
  → 技术架构师（系统设计）
  → 安全架构师（安全模型）
  → 数据架构师（数据模型）

第 2 阶段：基础与脚手架
  → 基础设施工程师（部署管线）
  → 开发者工具工程师（开发环境）
  → 身份与访问工程师（认证/授权）

第 3 阶段：构建与迭代
  → 前端开发者（用户界面）
  → 后端工程师（API 和数据层）
  → 移动开发者（移动应用）
  → 数据工程师（数据管道）
  → 提示工程师（AI 集成）
  → 实时协作工程师（协作功能）
  → 语音 AI 集成工程师（语音功能）

第 4 阶段：质量与加固
  → 测试工程师（测试套件）
  → 安全审计师（安全审查）
  → 渗透测试师（渗透测试）
  → 可访问性专家（无障碍审计）
  → 性能工程师（性能优化）

第 5 阶段：上线与增长
  → DevOps 自动化师（部署管线）
  → 网站可靠性工程师（监控/告警）
  → 增长黑客（增长策略）
  → 营销战略师（营销计划）

第 6 阶段：运营与演进
  → 事件响应指挥官（安全事件）
  → SRE（运营）
  → 技术写作（文档化）
  → 现实检查者（持续质量监控）
```

### 目标

### 激活代理（并行执行）
| 代理 | 产出 | 时间 |
|------|------|------|
| 前端开发者 | 用户界面、组件库、响应式设计 | 持续 |
| 后端工程师 | API 服务、数据库集成、业务逻辑 | 持续 |
| 移动开发者 | 移动应用、离线支持、推送通知 | 持续 |
| 数据工程师 | 数据管道、ETL 流程、分析仪表板 | 持续 |
| 提示工程师 | AI 集成、提示优化、模型选择 | 持续 |
| 实时协作工程师 | 协作功能、冲突解决、实时同步 | 持续 |
| 语音 AI 集成工程师 | 语音识别、语音合成、多语言支持 | 持续 |

### Dev↔QA 循环协议
```
1. 开发者完成功能 → 提交代码
2. 测试工程师编写测试 → 验证功能
3. 测试通过 → 进入下一阶段
4. 测试失败 → 开发者修复 → 回到步骤 1
5. 最大重试次数：3 次
6. 超过 3 次 → 升级到编排器或人
```

### 质量关卡（每轮迭代）
- [ ] 功能实现完成：通过代码审查
- [ ] 测试覆盖率：至少 80% 代码覆盖率
- [ ] 无严重 bug：所有高优先级问题已解决
- [ ] 集成通过：所有模块协同工作

## 12. 质量关卡

### 关卡定义

| 关卡 | 检查点 | 通过标准 | 失败动作 |
|------|--------|----------|----------|
| 关卡 0 | 规格完整性 | 用户故事有验收标准 | 返回阶段 0 |
| 关卡 1 | 架构决策 | 架构图 + 技术栈决策 | 返回阶段 1 |
| 关卡 2 | 基础设施就绪 | CI/CD 运行 + 开发环境可用 | 返回阶段 2 |
| 关卡 3 | 功能完成 | 测试通过 + 代码审查 | Dev↔QA 循环 |
| 关卡 4 | 质量保障 | 安全/性能/无障碍通过 | 返回阶段 4 |
| 关卡 5 | 上线就绪 | 部署完成 + 监控运行 | 返回阶段 5 |

### 关卡执行流程

```
1. 代理完成产出物
2. 代理提交质量关卡申请
3. 现实检查者验证：
   a. 产出物完整性检查
   b. 验收标准对照
   c. 证据审查
4. 通过 → 推进到下一阶段
5. 失败 → 返回代理修复，重新提交
```

### 关卡失败处理

- 第一次失败：代理修复，重新提交
- 第二次失败：代理修复，现实检查者指导
- 第三次失败：升级到编排器，人介入

## 14. 成功指标

### 定量指标

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 时间线压缩 | 40-60% | 并行 vs 顺序执行对比 |
| 质量关卡通过率 | > 95% | 一次通过 vs 需要重做 |
| 交接成功率 | > 97% | 无上下文丢失的交接 |
| 缺陷捕获率 | > 95% | 阶段内捕获 vs 集成后发现 |
| 代理利用率 | > 80% | 活跃工作 vs 等待时间 |
| 最终产品满意度 | > 4.5/5 | 用户反馈评分 |

### 定性指标

- 代理协作顺畅，无重复工作
